### イベントへの応答

まあこれはJSでできる内容でReactではどうするかっていう話かな。  

> <button> のような組み込みコンポーネントは onClick のような組み込みのブラウザイベントのみをサポートします

確かに、`onClick`はブラウザサポートされているよね。  
ただ、独自コンポーネントはどうやるのかという話。  

で、以下のようなコンポーネントがあると。  


```js
export default function App() {
  return (
    <Toolbar
      onPlayMovie={() => alert('Playing!')}
      onUploadImage={() => alert('Uploading!')}
    />
  );
}

function Toolbar({ onPlayMovie, onUploadImage }) {
  return (
    <div>
      <Button onClick={onPlayMovie}>
        Play Movie
      </Button>
      <Button onClick={onUploadImage}>
        Upload Image
      </Button>
    </div>
  );
}

function Button({ onClick, children }) {
  return (
    <button onClick={onClick}>
      {children}
    </button>
  );
}

```

これは、関数をpropsとして渡されていると。  
ちなみに、関数に引数がありそれを渡したい場合はラムダで包むのが良くあるパターンらしい。
例えば、onPlayMovieに引数を持たせたい場合は、

```js
↓コンポーネント側
    <Toolbar
      onPlayMovie={(t) => alert(t + ' ' + 'Playing!')}
      onUploadImage={() => alert('Uploading!')}
    />


↓コンポーネント呼び出し側
      <Button onClick={ () => onPlayMovie("arg1") }>
        Play Movie
      </Button>
```

とすればよいとのこと。

よくあるミスというより自分もやったミスとしては、以下のようなコードである。  

```js
      <Button onClick={onPlayMovie("aaaa")}>
        Play Movie
      </Button>
```

これだとレンダリング時に実行されてしまうので注意。  


### state：コンポーネントのメモリ

> コンポーネントによっては、ユーザ操作の結果として画面上の表示内容を変更する必要があります。

- フォーム上でタイプすると入力欄が更新される、画像カルーセルで「次」をクリックすると表示される画像が変わる。  
- 「購入」をクリックすると買い物かごに商品が入る  

> コンポーネントは、現在の入力値、現在の画像、ショッピングカートの状態といったものを「覚えておく」必要があります。React では、このようなコンポーネント固有のメモリのことを state と呼びます。

### レンダーとコミット

> コンポーネントが料理人として厨房に立ち、食材を調理して美味しい料理を作っている様子をイメージしてみてください。このシナリオにおいて React はウェイターです。お客様の注文を伝えて、できた料理をお客様に渡します。この UI の「注文」と「提供」のプロセスは、次の 3 つのステップからなります：

1. レンダーのトリガ（お客様の注文を厨房に伝える）
1. コンポーネントのレンダー（厨房で注文の品を料理する）
1. DOM へのコミット（テーブルに注文の品を提供する）

### state はスナップショットである

> 通常の JavaScript の変数とは異なり、React の state はスナップショットのような動作をします。セットしても既存の state 変数は変更されず、代わりに再レンダーがトリガされます。初見ではびっくりするかもしれません。

ほほう。
ちょっとサンプルコードを見てみる。  

```js
console.log(count);  // 0
setCount(count + 1); // Request a re-render with 1
console.log(count);  // Still 0!
```

あーこれはまだゼロなんだね。  

> このおかげで、見逃しやすい小さなバグを回避することができます。ここに小さなチャットアプリがあります。まず「送信」を押して、次に受信者を「ボブ」に変更したら何が起こるか、推測してみてください。5 秒後の alert には誰の名前が表示されるでしょうか？

なるほどね。どうやら次のレンダリングが走って初めて変わるらしい。  


ちょっと気になるのでClaudeに聞いたところメモ

```txt
setCount を呼んだ直後にReactがスケジュールして、現在の処理が終わったあとにまとめてレンダリングされます。

例えば：
jsxsetCount(count + 1)
setCount(count + 1)
setCount(count + 1)

これを3回呼んでも1回分しか増えないんです。全部「次は1にしてね」という同じお願いになるので。
これをまとめて処理する仕組みをバッチングと言います。

もし3回分増やしたいなら関数形式で書く必要があります：
jsxsetCount(c => c + 1)
setCount(c => c + 1)
setCount(c => c + 1)

この形だと「直前の値に+1して」という命令になるので、ちゃんと3増えます。
```

### 一連の state の更新をキューに入れる

あーこれはさっき気になって調べたやつそのままだ。  

`score`で参照している値がそのレンダリング中はスナップショットになっているから、値が固定されているってことだよね。  
つまり、score = 1 だったとしたら、それが3回実行されても score + 1 = 2になるわけだ。  
実質的には1回しか実行されないということ。  

> state はスナップショットであるで、なぜこのようなことが起こってしまうのかを説明しています。state をセットすると、新しい再レンダーが要求されますが、すでに実行されているコード内の state は変更されません。そのため setScore(score + 1) を呼び出した直後は score が 0 であり続けます。

> state を設定する際に更新用関数を渡すことでこれを修正することができます。setScore(score + 1) を  setScore(s => s + 1) に置き換えることで、“+3” ボタンが修正されることに注目してください。こうすることで、複数回の state の更新がキューに入れられます。

### state 内のオブジェクトの更新

> state にはどのような JavaScript の値でも保持することができます。これにはオブジェクトも含まれます。

しかし、React の state に保持されたオブジェクトと配列を直接書き換えるべきではありません。

へぇーそうなんだ。  

> オブジェクトを更新したい場合、代わりに新しいオブジェクトを作成（または既存のもののコピーを作成）し、それを使って state をセットする必要があります。
